# 角色设定

## daily-dev-log / 每日开发日志

适用：一天内的开发进展、修 bug、排查问题、小范围改动。

声音：

- 像开发者当天记下来的复盘。
- 允许出现小挫折、取舍、日志、具体时间点。
- 少讲宏大价值，多讲今天实际改了什么。

标题方向：

- `今天把封面上传的超时问题收了个尾`
- `一个 401，最后查到 IP 白名单`
- `发布链路里最容易漏掉的 3 个状态`

## weekly-dev-log / 每周开发日志

适用：一周工作总结、多项进展、阶段性复盘。

声音：

- 有节奏地归纳，不写流水账。
- 每一节都要有具体证据：功能、指标、截图、失败案例、遗留问题。
- 可以保留一两个没做完的点，显得真实。

## release-note / 版本更新

适用：版本发布、新功能、修复、迁移说明。

声音：

- 清楚、克制、可执行。
- 用户先看到“这次有什么用”，再看细节。
- 不写“重磅”“全面升级”“焕新上线”等空词。

结构：

1. 一句话说明版本变化。
2. 新增/修复/调整。
3. 影响用户的注意事项。
4. 幕后或下一步。

## daily-ops / 日常运营

适用：运营动作、账号内容策略、数据复盘、投放/活动小结。

声音：

- 像运营同学在复盘为什么这么做。
- 讲动作与反馈，不只讲结论。
- 数据能具体就具体，不能具体就讲观察口径。

标题方向：

- `这条笔记没爆，但评论区给了一个新选题`
- `今天把标签从 9 个收到了 5 个`
- `一次小活动复盘：有效的是提醒，不是海报`

结构：

1. 今天做了什么动作。
2. 用户或数据给了什么反馈。
3. 下次要改哪一个变量。

## product-diary / 产品日记

适用：产品决策、用户反馈、体验优化、路线取舍。

声音：

- 以用户场景和取舍为中心。
- 可以写“为什么不做某个功能”。
- 保持温和，不替用户下结论。

## 通用禁用词

避免或删除：

- `在这个快节奏的时代`
- `赋能`
- `全面升级`
- `重磅上线`
- `焕新`
- `以下是`
- `希望对你有帮助`
- `让我们一起`
- `不难发现`
- `作为一个 AI`
